01 · Background

What is Google Search Console Regex?

GSC's Performance report has a default filter that does contains, equals, and not contains. Useful for one URL or one keyword. Useless when you want to ask the report something like "all product pages except the legacy ones, in the UK, from organic, in the last 28 days".

The Matching regex option in the filter dropdown is what unlocks that. Once you switch to regex, you can describe URL or query sets that the standard filter can't reach — and you can do it in one filter row instead of stacking five.

The catch: GSC uses Google's RE2 implementation, not PCRE. Most of what you've memorised from JavaScript or Python does work. A handful of patterns (lookbehinds, backreferences, recursion) do not. The cheat sheet below lists what's safe.

03 · Copy-paste patterns

Ready-made GSC regex patterns

Six patterns that cover ~80% of real GSC analysis. Paste, swap the placeholders, ship.

URLs under a subfolder Pages filter
^/products/.*$

Anchored to the path. Swap products for your subfolder. The trailing .*$ catches everything below.

Brand queries (case-insensitive) Queries filter
(?i)(magstags|mags\s+tags|magtags)

(?i) turns on case-insensitive matching. List your brand spellings, including the common typos, separated by |.

Non-brand queries (inverse) Queries filter · Doesn't match
(?i)(magstags|mags\s+tags|magtags)

Same pattern, different dropdown. Set the filter to "Queries doesn't match regex" to subtract brand traffic from total.

Commercial-intent queries Queries filter
(?i)(buy|price|cheap|review|near\s+me|best)

Catches the high-intent modifiers people add. Add or remove tokens to match your vertical's signals.

Question queries Queries filter
^(how|what|why|when|where|who|which|is|are|does|can)\s.*$

Starts-with match on the canonical English question words. Useful for splitting informational vs transactional traffic.

URLs with no query string Pages filter
^[^?]*$

Matches paths that don't contain ?. Useful for separating clean canonical URLs from faceted/parameterised duplicates.

04 · RE2 syntax reference

GSC regex cheat sheet

The RE2 syntax that actually works inside Search Console. Bookmark this; the official Google docs hide it across three help articles.

^
Start of string

Anchors the pattern to the beginning.

^/blog/
$
End of string

Anchors the pattern to the end.

.html$
.*
Any characters

Matches zero or more of anything.

^/uk/.*$
|
OR

Either side matches.

buy|shop
(...)
Group

Groups for alternation or quantifiers.

(uk|us|ca)
[abc]
Character class

Any one of the listed characters.

[0-9]
(?i)
Case-insensitive

Inline flag at the start of the pattern.

(?i)Black Friday
\s
Whitespace

Matches a space, tab, or newline.

near\s+me
\d
Digit

Matches 0–9.

2026|\d{4}
05 · Real-world scenarios

5 ways to use GSC regex in your SEO workflow

  1. Brand vs non-brand split. Two filters, same pattern, opposite dropdowns. Pulls organic traffic minus brand awareness — the actual SEO contribution.
  2. Audit a faceted-nav explosion. Filter pages to \? to count parameterised URLs receiving impressions. The number is usually larger than people expect.
  3. Find pages losing clicks to AI Overviews. Filter queries to question-starters, then sort by impressions ÷ clicks ratio. The big-impression / low-CTR pages are the answers Google's summarising elsewhere.
  4. Spot regional cannibalisation. Filter pages to ^/(uk|us|au)/.*$, then group queries by country. Finds the cases where /us/ pages rank in UK SERPs (or vice versa).
  5. Track post-migration redirects. Filter pages to the old subfolder. After redirects ship, the impression count there should bleed toward zero over 4–8 weeks. If it doesn't, your redirects aren't being followed cleanly.
06 · Know the limits

What GSC regex cannot do

Four things that look like they should work but don't. Avoid wasting an afternoon.

!
No lookbehind

RE2 supports lookahead in some implementations, but GSC's variant drops (?<=...) and (?<!...). Restructure with anchors instead.

!
No backreferences

\1, \2, etc. are not supported. Match each segment independently if you need them.

!
Pattern length cap

GSC truncates patterns over ~4,096 characters silently. Long alternations get cut off — split into multiple filters or rebuild with character classes.

!
Position-aware filtering

Regex matches against the full URL or query, but you can't filter by "queries where position > 10". Combine with the position column manually after export.

07 · Common questions

Frequently asked questions

Does GSC regex support PCRE features like lookbehind?
No. GSC uses Google's RE2 implementation, which trades some PCRE features for guaranteed linear-time matching. Lookbehind, backreferences, and recursion are not supported. Most everything else works.
How do I do a case-insensitive match in GSC regex?
Prefix the pattern with (?i). For example, (?i)black friday matches "Black Friday", "BLACK FRIDAY", "black FRIDAY", etc.
Why does my long pattern stop working?
GSC silently truncates patterns over ~4,096 characters. Common cause: huge alternation lists. Either split into multiple filters or condense with character classes ([a-z] instead of a|b|c|d|...).
Can I use regex on dimensions other than Page and Query?
No. Regex matching is only available on the Page and Query dimensions. For Country, Device, and Search Appearance, use the default filter dropdowns.
Why doesn't my regex work in the Indexed Pages report?
The Performance report supports regex. Coverage, Sitemaps, and the Indexed Pages report use different filtering and do not support full regex — usually just substring matching.
