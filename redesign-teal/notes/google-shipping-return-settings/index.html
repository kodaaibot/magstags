Google Shipping and Return Settings: All Your Options, and What Takes Precedence

28 May 2026 12 min read By Mags Sikora

Shipping and return policies can be configured in multiple places with Google, and the interactions between those places are not always obvious.

The reason this gets complicated is that Google surfaces shipping and return information across multiple products (Shopping ads, free listings, product rich results), and each product has slightly different data requirements and ingestion paths. There is no single "shipping settings" field. There are five distinct configuration layers, each with its own scope and authority.

The five configuration layers

From most authoritative to least, here is the precedence ladder Google walks when it needs a shipping or returns answer:

PRECEDENCE.ORDER 
# Highest authority first
1. Merchant Center feed       → per-item, per-market
2. Merchant API               → programmatic overrides
3. Merchant Center settings   → account-level defaults
4. Search Console settings    → auto-syncs to MC
5. Schema markup              → page-level fallback
                              (Organisation schema = weakest)

The fact that all five can co-exist is the source of nearly every issue. Google does not aggregate them, it picks the first one it finds. If the per-item feed value disagrees with the schema markup on the page, the feed wins and the markup is silently ignored.

Shipping and returns is a source-of-truth problem, not a markup problem. The job is to pick which of the five layers is the canonical source for your setup, make every other layer agree with it, and accept that the rest are either inherited or quietly ignored.

The silent failure mode

The pattern most teams discover the hard way: a developer adds OfferShippingDetails to the product schema. The Rich Results Test passes. The page validates. Six weeks later, Merchant Center is still showing the old rate because the feed value (set up two years ago by someone who has since left) still has authority. Nothing is "broken" — but the change you shipped did not change anything visible to Google.

This is the failure that costs teams the most because there is nothing to debug. The validator is green. The schema is correct. The data underneath simply lost the argument.

Most product schema isn't broken. It's valid. The data underneath it just doesn't match the page, the feed, and the catalogue — and Google trusts none of them.
— from When Your Schema Is Valid But Your Data Isn't

Where each layer is the right source of truth

The right answer depends on scale and team structure. Here is when each layer typically becomes the canonical source:

Layer Use it when Avoid when
Feed Per-product rates vary; multi-market catalogue Rates are uniform across the catalogue
Merchant API Rates change frequently; programmatic pricing No engineering bandwidth for integration
MC settings Account-wide defaults; single-market Need per-item granularity
Search Console Small site without a feed; no MC account Multi-market; complex rate structures
Schema Only option (no MC/SC); explicit per-page fallback Anything authoritative exists upstream

Beyond Google: the durable principle

The precedence ladder above is specific to Google. The principle underneath it — alignment between feed, schema, and front-end — outlasts any specific platform. Other automated buyers, including AI shopping agents, compare the same layers, and when the layers disagree the rational response is to trust none of them. Consolidating to one canonical source is not a Google-only fix. It is what keeps the catalogue legible to whichever system reads it next.
